# PTO-DSL GEMM Kernel

## Overview

This directory contains a shape-specialized GEMM kernel written with PTO-DSL,
plus host-side scripts for building, validating, and benchmarking generated
libraries. `run.py` creates one generated kernel per case under
`case_builds/<case>/` and rebuilds it when the recorded build signature no
longer matches the selected configuration.

## Supported AI Processors

- A2/A3

## Directory Layout

```text
kernels/python/gemm/
├── kernels/
│   └── gemm_builder.py      # PTO-DSL source that emits PTO IR
├── compile.sh               # Builds build_artifacts/gemm.{pto,cpp,so}
├── caller.cpp               # Shared-library launcher wrapper
├── run.py                   # Correctness and benchmark runner
├── build_artifacts/         # compile.sh output for build() defaults
└── case_builds/             # Per-case generated kernels from run.py
```

Generated files are written under:

```text
build_artifacts/gemm.pto       # IR generated by compile.sh
build_artifacts/gemm.cpp       # C++ generated by ptoas
build_artifacts/gemm.so        # Shared library generated by compile.sh
case_builds/<case>/gemm.pto    # IR specialized for one run.py case
case_builds/<case>/gemm.cpp
case_builds/<case>/gemm.so
case_builds/<case>/build_config.json
```

## Operator Contract

The kernel computes:

```text
C = A x B
```

- `A`: `[m, k]`, `float16`, ND layout
- logical `B`: `[k, n]`, `float16`
- GM storage for `B`: `[n, k]`, passed as a transposed contiguous tensor and read with `layout="DN"`
- `C`: `[m, n]`, `float16`, ND layout
- accumulation: fp32 ACC tile, converted to fp16 on store

The Python runner creates logical `B[k, n]`, passes `B.T.contiguous()` to the
kernel, and uses the logical `B` tensor for correctness computation.

## Kernel Structure

`kernels/gemm_builder.py` emits one `Gemm` symbol specialized for the selected
shape and tiling. The implementation uses:

- persistent traversal over the global `baseM x baseN` output-tile grid
- N-group swizzling over base tiles to improve B-tile reuse in L2
- double-buffered L1 MAT staging tiles for A and B
- ping-pong L0A/L0B tile buffers
- one fp32 L0C ACC tile per output tile
- K-panel processing with panel width `baseK * stepKa`
- DN GM tensor view for the transposed B storage

The builder and runner validate these constraints:

- `m % baseM == 0`
- `n % baseN == 0`
- `k % baseK == 0`
- `stepKa == stepKb == 4`
- `(k / baseK) % (stepKa * 2) == 0`
- `1 <= blockDim <= (m / baseM) * (n / baseN)`
- `1 <= swizzleCountN <= n / baseN`

## Runtime Scheduling

Cases describe only shape and base tiling. At runtime `run.py` adds:

- `blockDim`: `min(cube_core_cnt, total_base_tiles)` from the selected SoC config
- `swizzleCountN`: an L2-budgeted N-group width capped by the configured target count

`--soc` can select a specific CANN platform config. If omitted, `run.py` uses
SoC environment variables when available and otherwise defaults to
`Ascend910B2`.

## build() Default Arguments

- `m = 6144`
- `k = 6144`
- `n = 6144`
- `baseM = 128`
- `baseK = 64`
- `baseN = 256`
- `stepKa = 4`
- `stepKb = 4`
- runtime `blockDim = cube_core_cnt` when the base-tile grid is large enough

## Prerequisites

Before building or running, ensure the following are available:

- Ascend CANN toolchain environment
- `bisheng`
- `ptoas`
- Python modules for MLIR/PTO (`mlir.ir`, `mlir.dialects.pto`)
- `torch` and `torch_npu` for `run.py`

`compile.sh` will try to source a local Ascend environment automatically when
`bisheng` is not already on `PATH`, and it also honors:

- `PTOAS_ROOT`: prepends PTO assembler binaries/libs to `PATH` and `LD_LIBRARY_PATH`
- `PTO_LIB_PATH`: overrides the repo root used for `include/`
- `NPU_ARCH`: passes the target NPU architecture to `bisheng`; defaults to `dav-2201`

## Build

Build the shared library from the `build()` default arguments:

```bash
cd ${git_clone_path}/kernels/python/gemm
bash compile.sh
```

For a different target architecture:

```bash
NPU_ARCH=dav-2301 bash compile.sh
```

Expected build steps:

1. Generate `build_artifacts/gemm.pto` from `kernels/gemm_builder.py`
2. Assemble PTO IR into `build_artifacts/gemm.cpp`
3. Compile `caller.cpp` into `build_artifacts/gemm.so`

## Run

List available cases:

```bash
cd ${git_clone_path}/kernels/python/gemm
python3 run.py --list-cases
```

Correctness test:

```bash
cd ${git_clone_path}/kernels/python/gemm
python3 run.py
```

Correctness test plus PTO benchmark:

```bash
python3 run.py --benchmark
```

Run optional Torch-NPU timing with the same ABt input storage:

```bash
python3 run.py --torch-npu
```

Both benchmark paths allocate A as `[m, k]` and B storage as `[n, k]`. The
Torch-NPU path calls `torch.matmul(a, b.transpose(0, 1))`. Timing uses NPU
hardware events and reports fp16-output latency and TFLOPS.

Run one case:

```bash
python3 run.py --case a2a3_perf_6144
```

Run all cases:

```bash
python3 run.py --all_cases --benchmark
```

Override the computed swizzle width:

```bash
python3 run.py --case a2a3_perf_6144 --swizzle-count-n 5
```

Force per-case rebuild:

```bash
python3 run.py --case a2a3_perf_6144 --rebuild
```

Use a specific shared library:

```bash
python3 run.py --case a2a3_perf_6144 --lib ./build_artifacts/gemm.so
```

`--lib` is valid only when exactly one case is selected, because the runner
still uses that case's shape and runtime launch parameters.

Select device with either of these environment variables:

- `PTODSL_TEST_DEVICE_ID`
- `TASK_DEVICE`

If neither is set, the runner defaults to `npu:0`.

## Expected Output

Correctness mode prints a `PASS` or `FAIL` line for the configured shape and
ends with:

```text
Result: ALL PASS
```

Benchmark mode additionally prints average latency and TFLOPS.
